# Verify publishing before and after the write

Publishing first rereads the detail bound to the edit session, aborts when that preflight observes a deleted configuration, and requires its content hash to match the edit baseline; a mismatch opens a remote-versus-draft diff with no P0 force-overwrite path. A matching baseline leads to a final diff and explicit target confirmation followed by one non-retried publish request: V1 supplies the baseline `casMd5` for server-side CAS and maps rejection to `WriteConflict`, while the public V3 Admin API has no CAS parameter and remains a supported P0 publish path without CAS parity or a separate product label. Its preflight cannot prevent a deletion or concurrent update in the interval before POST, and V3's internal verified state means only that the subsequent read-back observes the commanded result, not that no concurrent value was overwritten. Once the request may have left the client, cancellation or an ambiguous response enters server-state-unknown, immediately changes the retained pre-write detail from confirmed to unconfirmed, and forces a read-back using the original operation context and credential snapshot; another publish is blocked and the draft is retained until resolved. Read-back equal to the command's content and preservable metadata yields verified state; equality with the preflight baseline shows only that the command result is not currently visible, retains the draft, and returns to dirty with a fresh preflight required, without proving that the write never applied; a third value or deletion yields conflict and a diff. Read-back failure remains server-state-unknown and follows the normal typed cache visibility matrix, with reconciliation as the only retryable network action. Only verified state may cache the publish command's result and clear dirty, while every reconciliation read-back remains an authoritative observation ordered through the cache high-water and may independently refresh, delete, or block cached data.
