# Order cache mutations by observation

Every remote operation obtains an in-process monotonically increasing observation sequence when it starts, and every cache mutation carries that sequence with its captured context and exact cache coordinate or visibility-gate scope. Each coordinate, access identity, access-identity-plus-namespace, and `(AccessIdentity, Capability, optional scope)` gate maintains its own high-water: a later-started accepted observation wins even if an earlier request completes afterward, including authentication or permission blocks and the matching successes that clear them. Publish reconciliation read-backs obey this rule like every other remote read rather than being suppressed until the publish state becomes verified. A complete namespace index may remove an absent item only when no newer detail observation exists, partial or failed indexes never delete, an authoritative not-found records deletion, and a later-started successful detail read may restore a recreated configuration. Observation sequences reset on process start; persisted profile tombstones remain absolute and reject all mutations regardless of sequence.
