# Keep configuration history read-only and identity-local

P1 exposes history as an optional adapter capability with equivalent user flows for V1 and V3, not as a cross-generation dataset: every list, historical detail, and diff remains inside one complete access identity, canonical namespace, and configuration coordinate. Users may list revisions, open historical content, compare two historical revisions, or compare one revision with the current detail through IntelliJ Diff, but cannot restore, roll back, or republish a historical version. `CapabilityUnsupported`, `PermissionDenied`, and an authoritative empty history remain distinct states. History metadata and bodies use a bounded identity-keyed in-memory cache only; P1 persists no historical body and promises no offline history, while the ordinary current-detail cache keeps its separate existing policy.
