# Retire cache schemas whose ownership cannot be proven

The first release with versioned access identities and canonical namespaces stops reading every older persistent-cache schema because those entries cannot prove a profile ID, access revision, resolved API generation, or canonical namespace. Environment profiles and PasswordSafe credentials migrate separately, while obsolete list pages, indexes, and details are removed in bounded idempotent background work that never blocks startup; cleanup failure cannot reactivate the old reader. The first offline launch may consequently contain no configuration data, but the one-time upgrade summary explains that the legacy cache was deliberately retired instead of presenting an unexplained empty result. After a new-schema AUTO identity has been successfully resolved once, its separately persisted last-known generation may locate only that exact new-schema identity during a later offline restart, with every restored dataset marked unconfirmed until formal detection succeeds.
