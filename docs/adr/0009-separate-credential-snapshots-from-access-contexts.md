# Separate credential snapshots from access contexts

After any required AUTO bootstrap has resolved a generation, every ordinary formal project operation captures a safe immutable access context and resolves its secret exactly once from the PasswordSafe slot keyed by that context's profile ID and access revision into a separate in-memory credential snapshot; the ordinary operation is not considered captured until both pieces match. The earlier generation probe instead uses its own immutable probe context with the same revision-pinned credential material, and isolated diagnostics use the unsaved form snapshot without reading or mutating a persisted slot. Credential snapshots never participate in equality, hashing, logging, persistence, or cache keys, while login retries and token refreshes for that operation continue to use the captured snapshot; credential edits therefore affect only operations started afterward, and a missing current slot fails closed rather than reading an older revision.
