# Gate stale-cache visibility by failure type

Transport, rate-limit, server, and malformed-response failures may retain and show same-identity cached data with confirmation state `REFRESH_FAILED`, independently of whether its age is within TTL, stale, or deep-stale. A terminal adapter-mapped `AuthenticationFailed` only after any allowed recovery persistently blocks that identity's cache, while `PermissionDenied` from a configuration read blocks only that identity and namespace; permission failures from publish, namespace discovery, and optional capabilities affect only their respective capability and never hide authorized read data. Every block and matching success carries the operation's observation sequence through a scope-specific high-water, so an older late failure cannot reinstate a block after newer success and an older success cannot clear a newer block. A new access revision or the first newer matching success after restart may clear the scoped block; invalid configuration and unsupported generation never fall back to cache, an authoritative detail `ResourceNotFound` removes only that coordinate, and cancellation before a possible write changes neither the dataset nor its confirmation or age. An ambiguous write separately marks the publish outcome as server-state-unknown, preserves the previous detail but immediately changes its confirmation to `UNCONFIRMED`, and forces a read instead of caching the draft; a failed reconciliation follows the same typed visibility matrix, so an eligible retained detail becomes `REFRESH_FAILED` while terminal access failures apply their normal scoped block.
