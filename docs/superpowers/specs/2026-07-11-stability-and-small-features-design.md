# Nacos Search 稳定性加固与小功能设计

## 背景

Nacos Search 已具备多环境、namespace、远程搜索、本地缓存、认证、配置详情，以及 Java 代码与 Nacos 配置之间的导航能力。随着缓存持久化、并发内容拉取和 PSI 导航加入，插件的主要风险已从功能缺失转向 IDE 热路径阻塞、后台任务生命周期、网络兼容和缓存降级语义。

本设计覆盖未来一至两个版本。第一优先级是消除高风险故障与卡顿；新功能只选择能够复用稳定性底座、开发范围可控且用户可感知的项目。

## 目标与非目标

### 目标

- PSI 和 UI 热路径不等待网络、磁盘或长耗时协程。
- 插件、项目和 UI 组件销毁后，不残留所属任务与监听器。
- 统一 Nacos 请求的超时、取消、重试、认证刷新、错误分类和兼容策略。
- 网络失败时保留可用数据，并明确区分实时、新鲜缓存和陈旧缓存。
- 在稳定性底座完成后增加连接诊断、数据新鲜度提示和最近搜索。
- 所有 UI 功能在实现前提供精确改动位置和完整 UI 样例图，并取得用户确认。

### 非目标

- 不在本轮加入配置编辑、发布或回滚。
- 不加入配置变更监听或长轮询。
- 不实现跨环境配置 diff。
- 不引入云同步、复杂标签或收藏夹系统。
- 不做与稳定性和本轮小功能无关的大规模重构。

## 方案选择

采用“稳定性底座 + 小步功能”路线。

首个版本集中处理 EDT 隔离、生命周期、网络容错和缓存降级；后续版本基于统一状态与错误模型增加三个轻量功能。相比按用户链路逐条修补，该路线可以避免在搜索、详情和代码导航中重复实现容错逻辑；相比先建设完整可观测性，它更符合优先消除现有高风险故障的目标。

## 稳定性架构

### 后台任务边界

网络访问、缓存文件 I/O、全量配置加载和重解析必须在 EDT 之外执行。EDT 只消费已准备好的不可变状态并执行小规模 UI 更新。

所有长期任务必须绑定 IntelliJ `Disposable`：

- 应用级任务绑定应用服务。
- 项目搜索、PSI 预热和项目监听器绑定项目。
- 工具窗口与面板监听器绑定其 UI `Disposable`。
- 环境或 namespace 切换取消上一上下文的网络与加载任务。

`CacheService` 应具备明确的释放逻辑，销毁时取消内部 `CoroutineScope`。UI 组件不得创建无法追踪的独立 scope。

### PSI 热路径

`NacosKeyResolver`、行标记和引用解析属于 IDE 高频回调，应视为最高风险路径。它们只能读取快速内存快照，不得通过 `runBlocking` 等方式等待缓存完整加载或磁盘单项读取。

缓存未命中时返回“暂不可解析”，同时去重触发后台预热；数据就绪后通过 IntelliJ 合法的刷新机制重新计算标记或引用。后台预热失败不得阻塞代码分析，也不得持续高频重试。

### 搜索竞争与大 namespace

`NacosSearchService` 统一 debounce、立即搜索、翻页、刷新和上下文切换的取消关系。每个请求携带上下文身份与递增代次；只有当前环境、namespace 和最新代次仍匹配时才能发布结果。

全量配置加载保持有限并发，并增加：

- 总任务取消传播；
- 单项请求超时；
- 有界并发；
- 批次化状态发布；
- UI 侧增量或分页渲染，避免大量 Swing 模型更新集中发生。

## 网络执行与兼容

将 `NacosApiService` 中分散的 HTTP 细节收敛到统一请求执行器。登录、namespace、配置列表和配置详情共享超时、取消、日志脱敏和错误映射，但各 API 保留自己的响应解析器。

### 错误分类

内部错误模型至少区分：

- 连接超时；
- 读取超时；
- DNS 或连接失败；
- 认证失败；
- 限流；
- 确定性的客户端错误；
- 服务端错误；
- 响应协议或 JSON 不兼容；
- 用户取消。

错误向 UI 暴露可操作的简短信息，详细日志必须隐藏密码、Token、Authorization 头和敏感查询参数。

### 重试规则

- 仅幂等查询可自动重试。
- 对瞬时连接错误、读取中断、限流和部分 5xx 使用短退避，并限制尝试次数与总耗时。
- 认证失败只允许刷新 Token 后重放一次。
- 用户取消、确定性 4xx 与解析错误不自动重试。
- 重试期间持续检查协程取消，取消后不得继续请求或更新 UI。

### Nacos 与代理兼容

集中处理 Nacos 1.x/2.x 和常见代理差异，包括可选字段缺失、错误响应并非 JSON、字符集缺失和 chunked 传输异常。兼容分支必须由固定响应样本测试覆盖，不能通过无限重试掩盖协议问题。

## 缓存一致性与降级

缓存身份必须完整包含规范化服务器地址、namespace、Data ID、Group 和数据类型。环境或凭据上下文切换不得复用不匹配的请求结果。

搜索结果携带来源与抓取时间：

- `REMOTE`：本次远程请求返回；
- `CACHE`：TTL 内的新鲜缓存；
- `STALE_CACHE`：远程失败后显式采用的过期缓存。

降级顺序为：远程成功、新鲜缓存、带明确标识的陈旧缓存、可操作的错误。远程刷新失败时保留当前列表和详情，不清空界面。

缓存持久化采用临时文件写入后原子替换。单个损坏条目被隔离、记录并删除，不应导致整个缓存加载失败。容量和条目清理在后台分批执行，磁盘 I/O 不得在持有覆盖全部缓存的锁时进行。

当前进程内的 TTL 判断优先使用单调时间，持久化条目同时保存墙上时间，以便重启后恢复并处理系统时钟变化。陈旧数据只有在明确允许降级的查询中返回。

## 小功能设计

### 连接诊断

设置页为当前服务器配置增加“诊断连接”。诊断按顺序检查地址格式与连通性、可识别的 Nacos API 能力、认证、namespace 查询和配置列表查询，并展示每阶段状态、耗时和简化错误。

诊断运行在后台，可取消，切换服务器时自动取消旧诊断。提供可复制的脱敏报告，报告不得包含密码、Token、Authorization 头或完整敏感查询参数。

初步改动位置：

- `src/main/kotlin/com/nanyin/nacos/search/settings/NacosConfigurable.kt`：入口按钮及当前服务器上下文。
- 新的独立诊断服务：编排已有 API 能力并生成结构化步骤结果。
- 新的诊断弹窗或面板：只展示诊断状态，不直接持有网络逻辑。
- 中英文 message bundle：全部新增文案。

### 数据新鲜度提示

配置列表区域展示“实时数据”“本地缓存”或“陈旧缓存/离线”，并显示最近成功更新时间。手动刷新失败时保留当前内容，状态切换为陈旧或错误提示，并允许再次刷新。

初步改动位置：

- `src/main/kotlin/com/nanyin/nacos/search/ui/NacosSearchWindow.kt`：组合搜索状态与窗口级展示。
- `src/main/kotlin/com/nanyin/nacos/search/ui/ConfigListPanel.kt`：列表附近的紧凑状态区域。
- `src/main/kotlin/com/nanyin/nacos/search/services/NacosSearchService.kt`：公开来源、更新时间和刷新失败但保留数据的状态。
- 中英文 message bundle。

### 最近搜索

按环境保存少量最近搜索条件，条目包含 namespace、Data ID、Group 与最后使用时间。选择条目只恢复条件，不自动发起请求，避免环境切换或误操作带来的意外网络访问；用户确认搜索后才执行。

历史记录本地持久化、按服务器环境隔离、去重并限制为每个环境最多 10 条。用户可删除单条或清空当前环境记录。密码、Token 与配置内容不得进入历史记录。

初步改动位置：

- `src/main/kotlin/com/nanyin/nacos/search/ui/SearchPanel.kt`：搜索框入口、历史列表和恢复交互。
- 独立轻量历史服务或持久化组件：负责隔离、限量、去重和清除，不放入网络搜索服务。
- `src/main/kotlin/com/nanyin/nacos/search/ui/NacosSearchWindow.kt`：传递当前环境与 namespace 上下文。
- 中英文 message bundle。

## UI 审批硬门槛

连接诊断、数据新鲜度提示、最近搜索以及后续任何 UI 功能，在实现前必须单独提交以下材料：

1. 精确到现有组件和 Kotlin 文件的改动位置。
2. 入口、布局、键盘与焦点行为，以及环境和 namespace 切换后的状态变化。
3. 正常、加载、空、错误、离线/陈旧和内容溢出状态。
4. 接近 IntelliJ 实际尺寸和视觉语言的完整高保真 UI 样例图，不接受只覆盖局部控件的草图。
5. 用户对改动清单和样例图的明确批准。

在该批准完成前，不得编写对应 UI 实现代码。样例可以通过 brainstorming visual companion 展示并迭代，最终确认版本应保留为设计或实施计划的可追溯附件。

## 测试与验证

### 单元测试

- 错误分类、重试条件、认证刷新上限和取消传播。
- 缓存键规范化、环境隔离、TTL、陈旧降级、损坏条目和原子写恢复。
- 最近搜索的限量、去重、环境隔离、删除与敏感数据排除。

### 并发与生命周期测试

- debounce 与立即搜索竞争。
- 翻页、刷新、环境和 namespace 快速切换时旧结果不能覆盖新状态。
- 服务或项目销毁后任务停止，监听器解除。
- 全量加载取消后不再继续发布批次或写入不匹配上下文。

### IntelliJ 平台测试

- PSI 高频调用不进行阻塞 I/O。
- UI 状态发布发生在正确线程。
- `Disposable` 树回收 scope、监听器和弹窗任务。
- 大结果集更新不会在 EDT 执行无界工作。

### 网络兼容测试

使用可控假服务模拟连接超时、读取超时、断流、非 JSON 错误、401、429、5xx、字段缺失和异常 chunked 响应。插件验证至少覆盖构建目标 2024.3.5、当前稳定版和已配置的 2026.1 EAP。

## 版本划分

### 1.3.4：稳定性版本

- PSI 与 EDT 热路径隔离。
- 服务、项目和 UI 生命周期治理。
- 搜索请求代次与取消统一。
- 统一网络错误、超时、重试和兼容策略。
- 缓存原子写、损坏隔离与陈旧降级模型。
- 数据来源的内部状态建模；若涉及可见 UI，仍须先经过 UI 审批门槛。

### 1.4.0：小功能版本

- 连接诊断。
- 数据新鲜度提示。
- 最近搜索。

三个功能分别通过 UI 审批门槛后实施；不要求在同一提交中同时完成。

## 验收标准

- PSI 和 UI 热路径不出现网络或磁盘等待。
- 取消或上下文失效后的请求不能更新 UI。
- 网络失败不会清空已有搜索结果或详情。
- 陈旧缓存始终有明确的数据来源标识。
- 服务和项目销毁后不存在所属后台任务或监听器。
- 自动重试符合幂等性、次数和总耗时限制。
- 单个损坏缓存条目不会破坏整个缓存加载。
- 所有 UI 功能均在实现前具备明确改动位置、完整状态样例图和用户批准记录。
