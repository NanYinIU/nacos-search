# Nacos Search 稳定性加固与小功能设计

## 背景

Nacos Search 已具备多环境、namespace、远程搜索、本地缓存、认证、配置详情，以及 Java 代码与 Nacos 配置之间的导航能力。随着缓存持久化、并发内容拉取和 PSI 导航加入，插件的主要风险已从功能缺失转向 IDE 热路径阻塞、后台任务生命周期、网络兼容和缓存降级语义。

本设计覆盖未来一至两个版本。第一优先级是消除高风险故障与卡顿；新功能只选择能够复用稳定性底座、开发范围可控且用户可感知的项目。

## 目标与非目标

### 目标

- PSI 和 UI 热路径不等待网络、磁盘或长耗时协程。
- 插件、项目和 UI 组件销毁后，不残留所属任务与监听器。
- 统一 Nacos 请求的超时、取消、重试、认证刷新、错误分类和兼容策略。
- 网络失败时保留可用数据，并明确区分实时、新鲜缓存和陈旧缓存。
- 在稳定性底座完成后增加连接诊断、数据新鲜度提示和最近搜索。
- 所有 UI 功能在实现前提供精确改动位置和完整 UI 样例图，并取得用户确认。

### 非目标

- 不在本轮加入配置编辑、发布或回滚。
- 不加入配置变更监听或长轮询。
- 不实现跨环境配置 diff。
- 不引入云同步、复杂标签或收藏夹系统。
- 不做与稳定性和本轮小功能无关的大规模重构。

## 方案选择

采用“稳定性底座 + 小步功能”路线。

首个版本集中处理 EDT 隔离、生命周期、网络容错和缓存降级；后续版本基于统一状态与错误模型增加三个轻量功能。相比按用户链路逐条修补，该路线可以避免在搜索、详情和代码导航中重复实现容错逻辑；相比先建设完整可观测性，它更符合优先消除现有高风险故障的目标。

## 稳定性架构

### 后台任务边界

网络访问、缓存文件 I/O、全量配置加载和重解析必须在 EDT 之外执行。EDT 只消费已准备好的不可变状态并执行小规模 UI 更新。

所有长期任务必须绑定 IntelliJ `Disposable`：

- 应用级任务绑定应用服务。
- 项目搜索、PSI 预热和项目监听器绑定项目。
- 工具窗口与面板监听器绑定其 UI `Disposable`。
- 环境或 namespace 切换取消上一上下文的网络与加载任务。

不再运行周期性自动全量刷新。用户通过前台手动刷新明确请求更新；namespace 切换和 PSI 允许按下述触发规则执行一次性按需预热，它们不属于周期性自动刷新。

`CacheService` 应具备明确的释放逻辑，销毁时取消内部 `CoroutineScope`。UI 组件不得创建无法追踪的独立 scope。

### PSI 热路径

`NacosKeyResolver`、行标记和引用解析属于 IDE 高频回调，应视为最高风险路径。它们只能读取快速内存快照，不得通过 `runBlocking` 等方式等待缓存完整加载或磁盘单项读取。

配置引用采用三态：`RESOLVED` 表示已在当前访问身份和 namespace 中确认存在；`UNRESOLVED` 表示成功检查后确认不存在；`UNAVAILABLE` 表示因数据未就绪、网络或认证问题、或操作取消而无法判断。只有 `UNRESOLVED` 才显示明确的未解析警告，`UNAVAILABLE` 不得伪装为配置缺失。

缓存未命中时返回 `UNAVAILABLE`，同时去重触发后台预热；数据就绪后通过 IntelliJ 合法的刷新机制重新计算标记或引用。后台预热失败不得阻塞代码分析，也不得持续高频重试。若未来为 `UNAVAILABLE` 增加中性图标或提示，必须先通过 UI 审批硬门槛。

陈旧缓存可以提供缓存导航目标，允许用户离线查看旧配置内容，但引用状态保持 `UNAVAILABLE`，不得据此宣称配置当前存在。任何 tooltip、弹窗或中性图标必须明确数据来自陈旧缓存，并先通过 UI 审批硬门槛。

### 搜索竞争与大 namespace

`NacosSearchService` 统一 debounce、立即搜索、翻页、刷新和上下文切换的取消关系。每个请求携带上下文身份与递增代次；只有当前环境、namespace 和最新代次仍匹配时才能发布结果。

全量配置加载保持有限并发，并增加：

- 总任务取消传播；
- 单项请求超时；
- 有界并发；
- 批次化状态发布；
- UI 侧增量或分页渲染，避免大量 Swing 模型更新集中发生。

全量加载的数据集完整性独立建模：`COMPLETE` 表示预期条目全部取得；`PARTIAL` 表示可靠列表已取得但部分详情失败；`FAILED` 表示连可靠列表也未取得。`PARTIAL` 可以展示成功条目和失败条目的元数据，并显示成功数与总数、允许失败项单独重试，但不得用于得出 `UNRESOLVED` 结论，也不得覆盖上一次完整 namespace 索引。

前台搜索严格受 15 秒操作预算约束。预算内完成全量索引则正常返回；超时且已有部分结果时停止前台任务并发布 `PARTIAL`。若存在仍在最大陈旧期内的上一次完整索引，则优先返回旧完整索引并标记 `STALE`，新部分结果只用于说明本次刷新进度，不替换旧完整索引。后台预热是独立、去重、可取消的任务，不继承超时前台任务继续运行；只有完整构建并原子发布的索引可被后续搜索消费。

namespace 索引过期本身不启动定时拉取。全量拉取只由以下事件触发，并按访问身份与 namespace 执行 single-flight 去重：

- 用户切换到索引缺失或过期的 namespace 时，一次性后台预热；
- 用户执行内容、正则或必须使用本地全量索引的通配符搜索时，按 15 秒前台预算拉取；
- 用户点击“刷新全部缓存”时，强制前台全量拉取；
- PSI 按下述冷却与暂停规则请求一次性后台预热。

普通 Data ID、Group 与分页搜索只刷新对应列表页，单个配置详情只刷新自身；两者都不触发全量拉取。环境、访问身份或 namespace 切换会取消已经失效的全量任务。

PSI 代码分析发现 namespace 索引过期时永不等待网络。同一访问身份与 namespace 已有全量任务时只复用该任务状态；没有任务且距离上次尝试已超过 5 分钟冷却期时，允许触发一次后台预热。预热失败后冷却期内不再尝试，打开更多代码文件不能绕过冷却。认证失败后暂停 PSI 自动预热，直到用户修改连接配置、手动刷新或连接诊断成功。

周期性自动刷新不再触发全量拉取。现有持久化的自动刷新字段为兼容旧设置文件而保留读取能力，但运行时忽略；设置页中的自动刷新控件在对应 UI 方案获批后移除。

## 网络执行与兼容

将 `NacosApiService` 中分散的 HTTP 细节收敛到统一请求执行器。登录、namespace、配置列表和配置详情共享超时、取消、日志脱敏和错误映射，但各 API 保留自己的响应解析器。

### 错误分类

内部错误模型至少区分：

- 连接超时；
- 读取超时；
- DNS 或连接失败；
- 认证失败；
- 限流；
- 确定性的客户端错误；
- 服务端错误；
- 响应协议或 JSON 不兼容；
- 用户取消。

错误向 UI 暴露可操作的简短信息，详细日志必须隐藏密码、Token、Authorization 头和敏感查询参数。

### 重试规则

- 仅幂等查询可自动重试。
- 普通交互请求的连接超时为 3 秒、读取超时为 8 秒，最多额外重试一次，包含退避和认证刷新重放在内的整个操作预算为 15 秒。
- 后台预热使用 3 秒连接超时和 8 秒读取超时，不自动重试，等待下一次去重后的预热机会。
- 连接诊断是用户明确发起的长操作，整个诊断预算为 30 秒。
- 对瞬时连接错误、读取中断和部分 5xx 使用约 250ms 且带小幅随机抖动的单次退避，并确保不突破对应操作预算。
- `429` 仅在 `Retry-After` 存在且等待后仍不突破操作预算时重试一次。
- 认证失败只允许刷新 Token 后重放一次。
- 用户取消、确定性 4xx 与解析错误不自动重试。
- 重试期间持续检查协程取消，取消后不得继续请求或更新 UI。

### Nacos 与代理兼容

集中处理 Nacos 1.x/2.x 和常见代理差异，包括可选字段缺失、错误响应并非 JSON、字符集缺失和 chunked 传输异常。兼容分支必须由固定响应样本测试覆盖，不能通过无限重试掩盖协议问题。

## 缓存一致性与降级

缓存身份必须完整包含访问身份、规范化服务器地址、namespace、Data ID、Group 和数据类型。访问身份由稳定的环境配置 ID、认证模式和用户名组成；匿名访问也是独立身份，密码和 Token 不参与身份计算。环境、用户名或认证模式切换后不得展示旧访问身份留下的缓存，认证失败也不得降级到其他访问身份的数据。旧身份缓存可以保持不可见并等待后台容量回收。

展示数据集携带相互独立的来源、新鲜度与抓取时间。来源表示数据从哪里取得：

- `REMOTE`：远程响应；
- `CACHE`：本地缓存。

新鲜度表示插件能否确认数据仍与当前服务端状态一致：

- `FRESH`：来自最近一次成功请求，或来自仍在 TTL 内且未发生后续刷新失败的缓存；
- `STALE`：缓存已过 TTL，或用户要求刷新但请求失败后继续保留的既有数据；
- `UNKNOWN`：插件恢复了持久化数据，但本次运行尚未成功确认当前 Nacos 环境状态。

降级顺序为：远程成功、新鲜缓存、带明确标识的陈旧或未知数据、可操作的错误。远程刷新失败时保留当前列表和详情，不清空界面，并立即把当前展示数据集标记为 `STALE`，即使原缓存 TTL 尚未到期。

缓存持久化采用临时文件写入后原子替换。单个损坏条目被隔离、记录并删除，不应导致整个缓存加载失败。容量和条目清理在后台分批执行，磁盘 I/O 不得在持有覆盖全部缓存的锁时进行。

引入访问身份后，无身份维度的旧格式配置缓存不得猜测归属或继续展示，升级时一次性废弃。清理过程必须可中断、可重复；部分文件删除失败只记录并留待下次清理，不得阻塞插件启动。非配置内容且不跨权限边界的数据不因本迁移被清理。

当前进程内的 TTL 判断优先使用单调时间，持久化条目同时保存墙上时间，以便重启后恢复并处理系统时钟变化。陈旧数据只有在明确允许降级的查询中返回。

TTL 只决定何时需要刷新。陈旧数据的最大可用期固定为七天，从最近一次成功远程获取时间计算；超过七天后不得展示或用于缓存导航，并由后台清理。首版不提供用户可配置项。

TTL 分别作用于 namespace 索引、配置详情和列表页条目，三者首版继续共享同一个 TTL 设置值，但过期相互独立：

- 只有 namespace 索引的新鲜度决定是否需要全量拉取；
- 单个配置详情过期只触发该详情的按需拉取；
- 列表页过期只触发对应搜索条件与页码的重新请求；
- 配置详情或列表页过期不得连带使 namespace 索引失效；
- 完整全量刷新可以更新索引及成功取得的详情；
- 部分全量结果只能更新成功取得的详情，不得刷新 namespace 索引 TTL。

## 小功能设计

### 连接诊断

设置页为当前服务器配置增加“诊断连接”。诊断按顺序检查地址格式与连通性、可识别的 Nacos API 能力、认证、namespace 查询和配置列表查询，并展示每阶段状态、耗时和简化错误。

诊断基于用户发起时的当前设置表单快照运行，包括尚未 Apply 的值。诊断快照仅在本次任务内存在，不修改活动环境，不写配置缓存、namespace 状态或最近搜索；临时认证 Token 不进入全局 Token 缓存，并在诊断结束时丢弃。切换设置项、关闭设置页或重新诊断会取消旧任务。报告标明快照来自已保存或未保存设置。

诊断运行在后台，可取消。提供可复制的脱敏报告，报告不得包含密码、Token、Authorization 头或完整敏感查询参数。

初步改动位置：

- `src/main/kotlin/com/nanyin/nacos/search/settings/NacosConfigurable.kt`：入口按钮及当前服务器上下文。
- 新的独立诊断服务：接收诊断快照，使用隔离的认证与请求上下文编排已有 API 能力，并生成结构化步骤结果。
- 新的诊断弹窗或面板：只展示诊断状态，不直接持有网络逻辑。
- 中英文 message bundle：全部新增文案。

### 数据新鲜度提示

配置列表区域分别展示数据来源与数据集新鲜度，例如“远程来源 · 新鲜”“缓存来源 · 新鲜”“缓存来源 · 陈旧”或“缓存来源 · 未知”，并显示最近成功更新时间。手动刷新失败时保留当前内容，新鲜度切换为陈旧并显示错误提示，同时允许再次刷新。

初步改动位置：

- `src/main/kotlin/com/nanyin/nacos/search/ui/NacosSearchWindow.kt`：组合搜索状态与窗口级展示。
- `src/main/kotlin/com/nanyin/nacos/search/ui/ConfigListPanel.kt`：列表附近的紧凑状态区域。
- `src/main/kotlin/com/nanyin/nacos/search/services/NacosSearchService.kt`：公开来源、更新时间和刷新失败但保留数据的状态。
- 中英文 message bundle。

### 最近搜索

按环境保存少量最近搜索条件，条目包含 namespace、Data ID、Group 与最后使用时间。选择条目只恢复条件，不自动发起请求，避免环境切换或误操作带来的意外网络访问；用户确认搜索后才执行。

历史记录本地持久化、按访问身份隔离、去重并限制为每个访问身份最多 10 条。身份切换后不展示其他身份历史，切回原身份时可以恢复；“清空当前环境记录”只清除当前访问身份。旧版本没有访问身份归属的搜索历史不迁移。密码、Token 与配置内容不得进入历史记录。

初步改动位置：

- `src/main/kotlin/com/nanyin/nacos/search/ui/SearchPanel.kt`：搜索框入口、历史列表和恢复交互。
- 独立轻量历史服务或持久化组件：负责按访问身份隔离、限量、去重和清除，不放入网络搜索服务。
- `src/main/kotlin/com/nanyin/nacos/search/ui/NacosSearchWindow.kt`：传递当前环境与 namespace 上下文。
- 中英文 message bundle。

## UI 审批硬门槛

连接诊断、数据新鲜度提示、最近搜索以及后续任何 UI 功能，在实现前必须单独提交以下材料：

1. 精确到现有组件和 Kotlin 文件的改动位置。
2. 入口、布局、键盘与焦点行为，以及环境和 namespace 切换后的状态变化。
3. 正常、加载、空、错误、离线/陈旧和内容溢出状态。
4. 接近 IntelliJ 实际尺寸和视觉语言的完整高保真 UI 样例图，不接受只覆盖局部控件的草图。
5. 用户对改动清单和样例图的明确批准。

在该批准完成前，不得编写对应 UI 实现代码。样例可以通过 brainstorming visual companion 展示并迭代，最终确认版本应保留为设计或实施计划的可追溯附件。

## 测试与验证

### 单元测试

- 错误分类、重试条件、认证刷新上限和取消传播。
- 3 秒连接超时、8 秒读取超时、15 秒交互预算与 30 秒诊断预算。
- 诊断快照不会改变活动环境、全局 Token、配置缓存或 namespace 状态。
- 缓存键规范化、环境隔离、TTL、陈旧降级、损坏条目和原子写恢复。
- 固定七天最大陈旧期及超期数据不可见规则。
- Namespace 索引、配置详情和列表页的独立过期与刷新边界。
- 访问身份切换和认证失败不会暴露其他身份的缓存。
- 最近搜索的限量、去重、环境隔离、删除与敏感数据排除。
- 最近搜索的访问身份隔离与旧版无归属历史废弃。

### 并发与生命周期测试

- debounce 与立即搜索竞争。
- 翻页、刷新、环境和 namespace 快速切换时旧结果不能覆盖新状态。
- 服务或项目销毁后任务停止，监听器解除。
- 全量加载取消后不再继续发布批次或写入不匹配上下文。
- 部分详情失败时保留成功条目，但不覆盖完整索引或产生配置缺失结论。
- 前台全量搜索在 15 秒截止，半成品不会被后续搜索当作完整索引消费。
- PSI 预热遵守按访问身份与 namespace 的 single-flight、5 分钟冷却和认证失败暂停规则。
- 周期性自动刷新不会发起全量拉取，远程更新由用户手动刷新或明确的按需预热触发。

### IntelliJ 平台测试

- PSI 高频调用不进行阻塞 I/O。
- 数据不可用时配置引用进入 `UNAVAILABLE`，不会产生未解析误报。
- 陈旧缓存可用于离线导航，但不会把引用标记为 `RESOLVED`。
- UI 状态发布发生在正确线程。
- `Disposable` 树回收 scope、监听器和弹窗任务。
- 大结果集更新不会在 EDT 执行无界工作。

### 网络兼容测试

使用可控假服务模拟连接超时、读取超时、断流、非 JSON 错误、401、429、5xx、字段缺失和异常 chunked 响应。插件验证至少覆盖构建目标 2024.3.5、当前稳定版和已配置的 2026.1 EAP。

## 版本划分

### 1.3.4：稳定性版本

- PSI 与 EDT 热路径隔离。
- 服务、项目和 UI 生命周期治理。
- 搜索请求代次与取消统一。
- 统一网络错误、超时、重试和兼容策略。
- 缓存原子写、损坏隔离与陈旧降级模型。
- 数据来源的内部状态建模；若涉及可见 UI，仍须先经过 UI 审批门槛。

### 1.4.0：小功能版本

- 连接诊断。
- 数据新鲜度提示。
- 最近搜索。

三个功能分别通过 UI 审批门槛后实施；不要求在同一提交中同时完成。

## 验收标准

- PSI 和 UI 热路径不出现网络或磁盘等待。
- 取消或上下文失效后的请求不能更新 UI。
- 网络失败不会清空已有搜索结果或详情。
- 陈旧缓存始终有明确的数据来源标识。
- 部分数据集明确展示完整性，且不会被用于证明配置不存在。
- 服务和项目销毁后不存在所属后台任务或监听器。
- 自动重试符合幂等性、次数和总耗时限制。
- 普通交互操作最迟在 15 秒预算结束时给出结果或降级状态，连接诊断最迟在 30 秒预算结束。
- 单个损坏缓存条目不会破坏整个缓存加载。
- 超过七天最大陈旧期的数据不会被展示或用于导航。
- 无访问身份的旧格式配置缓存不会被展示，且迁移清理失败不会阻塞启动。
- 所有 UI 功能均在实现前具备明确改动位置、完整状态样例图和用户批准记录。
